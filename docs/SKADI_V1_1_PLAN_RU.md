# Skadi v1.1 Plan (RU)

Дата: 2026-06-04
Статус: draft
Назначение: зафиксировать рабочий план после стабилизации `v1`-ядра и до расширений `v2+`.

## 1. Идея релиза

`v1.1` должен быть не "большим новым языком", а аккуратным развитием уже понятного ядра:

- сохранить читаемость и каноничность синтаксиса `v1`,
- усилить практическую применимость для CLI/system-style сценариев,
- закрыть боли вокруг toolchain UX, диагностик и документации,
- добавить первый действительно полезный слой math/core без взрыва surface area.

Рабочая формула `v1.1`:

- меньше "новых форм";
- больше законченности, предсказуемости и удобства использования.

## 2. Цели v1.1

1. Довести `skadi-cli` до статуса основного пользовательского входа в toolchain.
2. Сделать диагностики и error-коды более цельными и стабильными для пользователя.
3. Добавить первый math/core slice, полезный для embedded/gamedev/system задач.
4. Подтянуть документацию и showcase-набор до уровня "можно показывать как продуктовую витрину".
5. Не ломать замороженную канонику `v1` без отдельного RFC.

## 3. Не входит в v1.1

- модульная система и `import`
- полноценные runtime-модели `on interrupt` / `on event`
- concurrency primitives (`run`, `wait`, `Link`)
- новый строгий контракт индексации вместо `fail-soft`
- крупная переработка memory model
- полноценные встроенные `Vec2/Vec3/Vec4` и `Matrix2D` как обязательная часть релиза
- радикальная смена канонического синтаксиса `v1`

Это можно готовить как RFC/экспериментальные ветки, но не как обязательный scope `v1.1`.

## 4. Основные треки работ

### Track A. Productization CLI

Цель:

- сделать `tools/skadi-cli` рекомендуемым способом работы со Skadi.

Задачи:

- стабилизировать команды `new/init/check/build/run/doctor/target list`;
- довести `format` хотя бы до минимально полезного состояния:

  - единые отступы,
  - переносы блоков,
  - нормализация пустых строк,
  - без изменения смысла программы;
- улучшить сообщения об ошибках build/run:

  - понятный источник ошибки (`Skadi frontend`, `C compiler`, `runtime execution`);
- привести шаблон проекта `tools/skadi-cli/templates/basic` к showcase-качеству;
- зафиксировать `skadi-cli` как основной путь в quick start и docs.

Критерии готовности:

- новый пользователь может сделать `new -> check -> build -> run` без ручного копания в репозитории;
- `doctor` даёт actionable hints по host/cross toolchain;
- `format` работает хотя бы на каноническом подмножестве `v1`.

### Track B. Diagnostics & Reliability

Цель:

- сделать поведение компилятора более объяснимым и стабильным на ошибках.

Задачи:

- унифицировать semantic/codegen/runtime-facing сообщения по тону и структуре;
- расширить покрытие диагностическими кодами там, где ещё остаются "просто строки";
- отделить:

  - parse-ошибки,
  - semantic-ошибки,
  - toolchain/build-ошибки;
- добавить regression-набор на тексты диагностик для критичных пользовательских сценариев;
- зафиксировать policy для style warnings:

  - когда это warning,
  - когда это hard error,
  - что считается canonical alias warning.

Критерии готовности:

- одинаковые классы ошибок оформлены единообразно;
- ключевые diagnostics не "плавают" между коммитами без осознанного решения;
- CI ловит регрессии по критичным diagnostic-shape кейсам.

### Track C. I/O and Runtime UX

Цель:

- улучшить практическую полезность `v1`-ядра без резкого расширения языка.

Задачи:

- согласовать следующий шаг после текущих `read/write/input/output/args/fs.*`;
- описать переходный дизайн:

  - что остаётся "простым v1 API",
  - что уходит в будущий stream/runtime слой;
- при необходимости добавить малый набор безопасных улучшений builtins без синтаксической революции;
- дополнить e2e сценарии реальными CLI/file workflows.

Критерии готовности:

- у I/O story есть понятное продолжение, а не набор несвязанных builtins;
- showcase-программы покрывают типовые file/args/fs сценарии;
- новые runtime helper-ы не ломают текущий memory contract.

### Track D. Math Core 1st Slice

Цель:

- дать Skadi минимальный math baseline, оправданный для embedded/gamedev/system задач.

Scope для `v1.1`:

- константы:

  - `PI`
  - `TAU`
  - `E`
  - `EPSILON`
- базовые функции:

  - `abs`
  - `min`
  - `max`
  - `clamp`
  - `floor`
  - `ceil`
  - `round`
- тригонометрия и корни:

  - `sin`
  - `cos`
  - `atan2`
  - `sqrt`
  - `root`
- конверсия углов:

  - `deg_to_rad`
  - `rad_to_deg`

Что сознательно не обещаем в `v1.1`:

- полноценные `Vec2/Vec3/Vec4` как встроенные типы,
- `Matrix2D` final syntax,
- SIMD/backend-specific optimization.

Задачи:

- оформить точный RFC scope для math first slice;
- добавить parser/semantic/codegen поддержку builtins;
- подключить `math.h` там, где нужно;
- покрыть минимум:

  - smoke tests,
  - semantic negative tests,
  - 1-2 e2e сценария прикладной математики.

Критерии готовности:

- math API полезен уже в реальных небольших программах;
- типы и ожидаемые возвращаемые значения явно документированы;
- нет "магических" незафиксированных преобразований.

### Track E. Docs, Showcase, Release Readiness

Цель:

- превратить текущий набор документов в последовательную карту проекта.

Задачи:

- обновить quick start под `skadi-cli` как главный путь;
- синхронизировать:

  - `SKADI_SYNTAX_STATUS.md`,
  - `TEST_COVERAGE_MATRIX.md`,
  - `docs/SKADI_PROJECT_OVERVIEW_RU.md`,
  - `SHOWCASE_PROGRAMS.md`;
- расширить showcase так, чтобы он отражал реальный стиль языка, а не только тестовые формы;
- оформить отдельную страницу "What is stable in v1.1".

Критерии готовности:

- новый участник проекта понимает состояние языка без чтения десятка разрозненных файлов;
- showcase совпадает с каноническим стилем и реальным поддерживаемым syntax surface;
- релиз можно сопровождать компактной release note.

## 5. Предлагаемые приоритеты

P0 для начала `v1.1`:

1. Track A: CLI productization
2. Track B: diagnostics stabilization
3. Track E: docs synchronization

P1:

1. Track D: math core first slice
2. Track C: I/O/runtime UX alignment

Логика приоритета:

- сначала делаем toolchain и повседневный UX устойчивыми;
- потом расширяем "полезную мощность" языка;
- не смешиваем большой UX-polish и большие синтаксические нововведения в один этап.

## 6. Вехи

### Milestone 1: v1.1-alpha

Фокус:

- `skadi-cli` как основной путь,
- единообразные diagnostics,
- обновлённые docs.

Готово, если:

- quick start ведёт через `skadi-cli`,
- `doctor/check/build/run` устойчивы,
- release docs не противоречат текущему состоянию репозитория.

### Milestone 2: v1.1-beta

Фокус:

- math first slice,
- расширенный e2e/showcase набор.

Готово, если:

- реализован и протестирован согласованный math baseline;
- есть минимум один showcase/bench сценарий, где математика выглядит органично для языка.

### Milestone 3: v1.1-rc

Фокус:

- финальная чистка UX,
- заморозка docs и тестовой матрицы.

Готово, если:

- нет известных критичных расхождений между docs, parser, semantic и codegen;
- CI зелёный на полном обязательном наборе;
- список deferred-фич после `v1.1` явно зафиксирован.

## 7. Правила принятия новых фич в v1.1

Новая фича входит в `v1.1`, только если:

1. она усиливает основной пользовательский сценарий;
2. не создаёт вторую конкурирующую форму без сильной причины;
3. укладывается в канонический showcase-стиль, закреплённый в `examples/example_meteostation.skd` и `SKADI_STYLE_PRINCIPLES.md`;
4. сопровождается parser + semantic + codegen + e2e/tests;
5. не размывает границу между `v1.1` и будущим `v2`.

## 8. Коротко: каким должен ощущаться v1.1

После `v1.1` пользователь должен ощущать Skadi так:

- язык уже достаточно цельный, чтобы писать небольшие утилиты и showcase-программы без ощущения "сырого прототипа";
- CLI не мешает, а помогает;
- ошибки понятны;
- базовая математика уже есть;
- при этом язык остаётся компактным и читабельным, а не расползается во все стороны сразу.
