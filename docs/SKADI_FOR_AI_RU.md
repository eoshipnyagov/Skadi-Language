# Skadi для AI и vibe coding

Назначение этого документа: быстро ввести языковую модель в рабочий режим для
генерации и правки кода на текущем подмножестве Skadi.

Это не полный справочник языка. Это короткий operational guide: что писать,
чего не выдумывать, на какие документы опираться и как не ломать текущий
`Skadi -> C` pipeline.

Если нужен полный и точный справочник, опирайтесь на:

- [Справочник языка](language-reference.md)
- [Статус синтаксиса](syntax-status.md)
- [Showcase-программы](showcases.md)
- [Каноническая матрица синтаксиса](../internal/syntax-canonical-matrix.md)

## 1. Короткая модель языка

Skadi в этом репозитории сейчас:

- компилируется по цепочке `Skadi -> C -> native exe`;
- имеет ограниченную, но хорошо тестируемую поверхность `v1/v1.1`;
- предпочитает канонический и предсказуемый синтаксис;
- не любит “догадки по аналогии” с Rust, Python, Go, TypeScript или Kotlin;
- требует использовать только реально поддержанные конструкции и builtins.

Главное правило для модели:

> Пиши не “как мог бы выглядеть язык”, а только как он реально реализован в
> этом репозитории сейчас.

## 2. Главные правила для генерации кода

Всегда:

- используйте только конструкции, которые уже есть в `language-reference`,
  `syntax-status` и showcase-программах;
- предпочитайте канонический стиль из syntax matrix;
- пишите маленькие, прозрачные функции;
- делайте типы и control-flow максимально явными;
- если задача решается без новых абстракций, не придумывайте новые абстракции.

Никогда:

- не придумывайте новые ключевые слова;
- не придумывайте новые builtins;
- не придумывайте generics, match-expression, exceptions, traits, classes,
  modules, imports, lambdas, async/await и macro-system;
- не предполагайте существование GC, borrow-checker или exception runtime;
- не используйте синтаксис, если вы не видели его в документации или тестах.

## 3. Канонический стиль, который стоит использовать по умолчанию

Предпочтительно писать так:

```skadi
new Int count = 0
new Bool ok = true
new Text title = "Skadi"

iterate items as item {
    output(item)
}
```

А не так:

```skadi
new bool ok = true

for item in items {
    output(item)
}
```

Пояснения:

- `Bool` и `Char` предпочтительнее, чем `bool` и `char`;
- `iterate collection as item` предпочтительнее, чем `for item in collection`;
- код должен выглядеть как showcase-код, а не как экспериментальный синтаксис.

## 4. Что модель должна считать “безопасной поверхностью”

Обычно безопасно опираться на:

- объявления `new x = expr` и `new Type x = expr`;
- `fn` и `danger fn`;
- `if / else if / else`;
- `while`, `loop`, `break`, `continue`;
- `iterate collection as item`;
- `when / is / else`;
- `struct`, поля, методы, `my.field`;
- `List`, `Text`, `Path`, числа, `Bool`, `Char`;
- builtins для `Text`, `List`, `fs.*`, `args`, `input/output/read/write`;
- math core из `v1.1`.

Если задача укладывается в этот набор, почти всегда лучше оставаться внутри него.

## 5. Что модель чаще всего пытается выдумать, но делать этого нельзя

Типичные ошибки моделей:

- писать `let` или `var` вместо `new`;
- писать `match`, `switch expr`, `elif`, `foreach`, `try/catch`;
- писать `enum` вместо `label`;
- использовать `Result<T, E>`, `Option<T>`, `Some/None`, `Ok/Err`;
- возвращать ошибки через несуществующий exception-style flow;
- использовать `import`, `package`, `namespace`, `module`;
- писать `[]Type`, `Vec<T>`, `Array<T>` вместо `Type List`;
- придумывать методы и builtins вроде `text.trim()`, `path.exists()`,
  `list.map()`, `list.filter()`, `print()`, `panic()`;
- использовать несуществующие литералы и формы типов.

Если хочется написать что-то из этого списка, нужно остановиться и
переформулировать решение через уже существующие возможности языка.

## 6. Базовые формы, которые стоит помнить

### Объявление

```skadi
new x = 10
new Int count = 10
new Text name = "Alice"
new Path root = "."
new i32 List xs = [1, 2, 3]
```

### Функции

```skadi
fn add(Int a, Int b) -> Int {
    return a + b
}
```

```skadi
danger fn load_text(Path path) -> Text {
    return read(path)
}
```

### Обработка ошибок

```skadi
label ErrorCode {
    Ok,
    NotFound,
    InvalidData
}

new Text content = read(path) on error {
    output("read failed")
    return error NotFound
}
```

Правила:

- `on error` используйте только там, где он реально поддержан;
- `return error Code` имеет смысл только в `danger fn`;
- первый вариант в `label ErrorCode` должен быть `Ok`.

### Цикл по коллекции

```skadi
iterate files as file {
    output(file)
}
```

### Структуры и методы

```skadi
struct Counter {
    Int value

    fn inc(Int delta) {
        my.value = my.value + delta
    }
}

new Counter c = {value = 10}
c.inc(5)
```

### `when`

```skadi
when code
is 0 {
    output("ok")
}
is 1 {
    output("warn")
}
else {
    output("unknown")
}
```

## 7. Рекомендуемые паттерны для vibe coding

Если модель пишет новый код, лучше выбирать такие паттерны:

- одна функция решает одну понятную задачу;
- имена простые и прикладные: `count`, `root`, `files`, `summary`, `ok`;
- вычисления разбиваются на промежуточные `new`-переменные;
- файловая работа идёт через `Path`, `fs.join`, `fs.list`, `fs.is_dir`, `read`,
  `write`;
- обработка ошибок делается локально и явно;
- вывод для отладки и демонстрации делается через `output(...)`;
- showcase-стиль важнее “умных” сокращений.

Пример хорошего направления:

```skadi
fn list_dirs(Path root) {
    new Path List entries = fs.list(root)
    iterate entries as entry {
        if fs.is_dir(entry) {
            output(entry)
        }
    }
}
```

## 8. Паттерны, которых лучше избегать

Лучше не писать:

- слишком большие функции с несколькими уровнями вложенности;
- сложные выражения, если их можно разбить на шаги;
- стиль “всё в одну строку”;
- неочевидные сокращения имён;
- код, который требует недокументированных runtime-предположений;
- псевдо-обобщённый код под несуществующие generics.

Для Skadi сейчас лучше “прямолинейно и предсказуемо”, чем “абстрактно и умно”.

## 9. Как выбирать между несколькими синтаксическими формами

Если доступны и alias, и canonical-форма, выбирайте canonical:

- `Bool`, а не `bool`;
- `Char`, а не `char`;
- `iterate ... as ...`, а не `for ... in ...`.

Если в существующем файле уже используется alias-форма, при небольшой локальной
правке можно сохранить стиль файла. Но в новых примерах и новых showcase-кусочках
лучше возвращаться к канонике.

## 10. Как модели проверять себя перед выдачей кода

Перед тем как считать ответ готовым, полезно пройтись по короткому чеклисту:

1. Все объявления сделаны через `new`?
2. Все используемые типы и builtins реально существуют?
3. Нет ли синтаксиса из других языков?
4. Не придуман ли новый способ обработки ошибок?
5. Использован ли канонический стиль там, где это возможно?
6. Можно ли показать аналог в showcase или в текущих тестах?

Если хотя бы на один из пунктов ответ “не уверен”, лучше упростить код.

## 11. Как модели править существующий Skadi-код

Если задача не “написать с нуля”, а править существующий файл:

- сначала сохраните структуру и стиль файла;
- не меняйте синтаксическую форму без причины;
- не расширяйте поверхностно язык “по месту”;
- не добавляйте новые runtime-допущения;
- если видите legacy-форму, не обязательно переписывать её без запроса;
- локальная безопасная правка почти всегда лучше, чем красивый большой рефакторинг.

## 12. На что ссылаться модели в сомнительных случаях

При неуверенности приоритет такой:

1. `tests/` как источник реально проверяемого поведения;
2. `docs/SKADI_SYNTAX_STATUS.md` как срез текущей поддерживаемой поверхности;
3. `docs/SYNTAX_CANONICAL_MATRIX_V1_RU.md` как источник канонической формы;
4. `benchmarks/` и `docs/SHOWCASE_PROGRAMS.md` как образцы живого кода;
5. `docs/SKADI_LANGUAGE_REFERENCE_RU.md` как подробный справочник.

То есть лучше ориентироваться на реализованное и протестированное поведение, чем
на “интуитивно красивый” синтаксис.

## 13. Готовый краткий prompt-блок для AI

Ниже блок, который можно копировать в системный или стартовый контекст:

```text
Пиши только валидный Skadi из текущего репозитория.
Не придумывай новые ключевые слова, builtins, типы и конструкции.
Используй канонический стиль: Bool/Char, iterate ... as ..., new Type x = expr.
Если не уверен в синтаксисе, опирайся на syntax-status, language-reference,
syntax-canonical-matrix и showcase-программы.
Предпочитай маленькие, явные, проверяемые решения без скрытой магии.
```

## 14. Готовый расширенный prompt-блок для codegen-задач

```text
Ты пишешь код на Skadi для текущего репозитория.
Работай только в рамках реально поддержанного подмножества языка.
Не используй синтаксис из Rust, Python, Go, TypeScript, Kotlin или C#.
Не придумывай generics, imports, classes, enums, exceptions, lambdas или async.
Для объявления переменных используй new.
Для канонического стиля предпочитай Bool/Char и iterate ... as ....
Для ошибок используй только существующий danger/on error/ErrorCode flow.
Для примеров и сомнительных мест смотри на tests и showcase-программы.
```

## 15. Что считать хорошим результатом от модели

Хороший Skadi-код, сгенерированный моделью, обычно:

- компактен;
- синтаксически узнаваем;
- похож на showcase и тестовые программы;
- не выдумывает язык заново;
- легко проходит `check`, `format`, `build`;
- не требует “догадаться”, что автор имел в виду.

Если код выглядит как смесь Skadi с другим языком, это плохой результат, даже
если идея задачи сама по себе разумная.
