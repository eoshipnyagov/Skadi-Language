# Что сознательно не поддерживается в `v1`

Этот документ фиксирует границы языка на этапе `v1`.

Смысл страницы простой: Skadi не пытается сейчас быть всем сразу. Мы держим
язык читабельным, предсказуемым и компактным, а спорные расширения откладываем
до отдельного RFC.

## Что пока сознательно вне `v1`

- generics и параметризация типов;
- decorators и attribute-based метапрограммирование;
- pipelines вида `a |> b |> c`;
- operator overloading;
- macros и compile-time expansion layer;
- implicit module auto-imports;
- pattern matching как обязательная языковая форма;
- complex typeclass / trait universe;
- multiple competing syntax forms для одной и той же операции;
- hidden sugar, которое нельзя явно прочитать из исходника.

## Почему это не входит сейчас

`v1` должен оставаться:

- компактным;
- легко читаемым;
- предсказуемым для parser/semantic/codegen;
- пригодным для рабочих утилит и системных сценариев без разрастания surface
  area.

Эти возможности не “запрещены навсегда”. Они просто не являются частью
релизного контракта `v1`.

## Что делать вместо этого

- если нужна универсальная композиция, использовать обычные функции;
- если нужен конфигурируемый стиль, держать его в явном code style и CLI/TUI
  настройках;
- если нужен более сложный синтаксический сахар, оформлять его отдельным RFC и
  не смешивать с core `v1`.

## Практическое правило

Если новая идея:

- добавляет вторую форму той же операции;
- ухудшает читаемость;
- делает parser/semantic контракт менее прозрачным;

то это почти наверняка не `v1`-фича.
