# Skadi v1: Контракт памяти C-runtime

Дата: 2026-05-25  
Статус: зафиксированный контракт для `v1` (`Skadi -> C`).

## 1. Цель

Этот документ фиксирует правила владения памятью для runtime helper-ов, которые генерирует транспилятор в C-коде.

## 2. Базовый принцип v1

- `v1` использует модель `process-lifetime` для части выделений:

  - память может не освобождаться вручную до завершения процесса;
  - это допустимо для CLI-утилит и короткоживущих программ `v1`.
- Приоритет `v1`: отсутствие крашей/UB и предсказуемое поведение.

## 3. Функции и владение

1. `sk_text_slice(...) -> char*`
- Выделяет новую строку через `malloc`.
- Владение: вызывающая сторона.
- В `v1` допускается не освобождать в короткоживущем процессе.

2. `sk_text_concat(...) -> char*`
- Выделяет новую строку через `malloc` (или `strdup("")` при fallback).
- Владение: вызывающая сторона.
- В `v1` допускается process-lifetime поведение.

3. `sk_read_file(...) -> char*`
- Возвращает буфер, выделенный через `malloc`/`strdup`.
- Владение: вызывающая сторона.

4. `sk_input(...) -> char*`
- Возвращает строку через `strdup`.
- Владение: вызывающая сторона.

5. `sk_fs_join(...) -> char*`
- Возвращает путь, выделенный через `malloc`/`strdup`.
- Владение: вызывающая сторона.

6. `sk_fs_list(...) -> SkadiList_text`
- Возвращает список; элементы списка выделяются через `strdup`.
- Владение:

  - контейнер `data` (через `realloc`) принадлежит списку;
  - каждый `char*` элемент принадлежит списку/владельцу списка.
- В `v1` освобождение элементов/контейнера явно не генерируется.

7. `sk_args(...) -> SkadiList_text`
- Аналогично `sk_fs_list`: элементы `strdup(argv[i])`.
- Владение: список/владелец списка.

8. `SkadiList_<T>`
- `push` может перераспределять `data` через `realloc`.
- `pop` не освобождает память, только уменьшает `len`.
- `get` не выделяет память.

## 4. Что гарантируется в v1

- Никаких скрытых `free` в runtime helper-ах, чтобы избежать double-free/висячих ссылок в текущей модели.
- Для поддерживаемых сценариев поведение детерминировано.
- Санитайзерный e2e-тест (`ASan/UBSan`) включен и должен проходить там, где компилятор поддерживает флаги.

## 5. Что НЕ гарантируется в v1

- Отсутствие утечек памяти в long-running процессе.
- Автоматическое освобождение всех runtime-аллоцированных объектов.
- Полноценная ownership/borrow модель языка на уровне рантайма.

## 6. План после v1

- Ввести явный слой освобождения для `Text`/`List<Text>` helper-объектов.
- Определить строгую политику `owning`/`borrowed` для runtime API.
- Расширить sanitizer-набор отдельными длительными стрессами.
