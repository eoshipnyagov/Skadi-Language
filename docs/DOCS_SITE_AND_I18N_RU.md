# Сайт документации и локализация (RU)

Этот документ фиксирует текущий технический подход к HTML-документации и
двуязычному контуру.

## 1. Выбранный стек

- `MkDocs`
- `Material for MkDocs`
- `mkdocs-static-i18n`

Почему так:

- текущая документация уже хранится в Markdown;
- нужен статический HTML-сайт без тяжёлой миграции;
- нужен нормальный путь к RU/EN;
- нужно сохранить docs-as-code workflow внутри репозитория.

## 2. Источник истины

Текущий источник истины:

- русские исходные документы в `docs/`

Под будущие английские переводы зарезервирован каталог:

- `docs-en/`

Сгенерированный сайт не является источником истины:

- он собирается в `.docs-build/` через sync-скрипт;
- итоговый HTML уходит в `site/`.

## 3. Структура сайта

Сайт делится на два больших раздела:

- пользовательские документы;
- внутренние документы разработки.

Это соответствует текущей логике проекта:

- пользователю не нужно читать draft-контракты memory/task/visual;
- разработчику компилятора не нужно продираться через onboarding-текст.

## 4. Как это собирается

Файлы:

- `mkdocs.yml`
- `requirements-docs.txt`
- `scripts/sync_docs_site.py`
- `scripts/clean_generated_en_docs.py`
- `scripts/docs_build.ps1`
- `scripts/docs_serve.ps1`
- `scripts/docs_build.sh`
- `scripts/docs_serve.sh`

Порядок работы:

1. `sync_docs_site.py` читает Markdown-источники из `docs/`
2. раскладывает их в `.docs-build/`
3. добавляет RU/EN страницы в формате suffix-mode
4. `mkdocs build` или `mkdocs serve` строит HTML-сайт

Для рабочего режима, когда нужно убрать сгенерированный английский слой и не
создавать ощущение, что править нужно "и там и там", есть отдельная очистка:

- `python scripts/clean_generated_en_docs.py`
- `scripts/clean_generated_en_docs.ps1`
- `scripts/clean_generated_en_docs.sh`

## 5. Почему выбран suffix-mode i18n

Причина простая:

- легче держать RU и EN страницы рядом;
- удобно постепенно добавлять английские версии;
- проще генерировать каркас для ещё не переведённых страниц.

Пример:

- `language-reference.md`
- `language-reference.en.md`

## 6. Политика перевода

На текущем этапе:

- `RU` является основным источником истины;
- `EN` развивается постепенно;
- при отсутствии EN-версии сайт показывает английскую заглушку.

Это лучше, чем:

- обещать полный EN-контур и не поддерживать его;
- публиковать сырые автопереводы без контроля.

## 7. Автоматический перевод

Рекомендуемый путь:

- не публиковать машинный перевод напрямую без проверки;
- использовать машинный перевод как черновик;
- ревьюить пользовательские страницы вручную.

Практически разумные варианты:

- `Weblate`
- `Crowdin`

Предпочтение по подходу:

- сначала пользовательские документы;
- затем выборочные внутренние документы;
- future drafts можно переводить позже или частично.

## 8. Что переводить в первую очередь

Первый обязательный EN-набор:

- `SKADI_GETTING_STARTED_RU.md`
- `SKADI_CLI_QUICK_START_RU.md`
- `SKADI_CLI_REFERENCE_RU.md`
- `SKADI_LANGUAGE_REFERENCE_RU.md`
- `SHOWCASE_PROGRAMS.md`

Второй слой:

- `SKADI_SYNTAX_STATUS.md`
- `docs/SKADI_PROJECT_OVERVIEW_RU.md`
- `SKADI_DOCS_USER_RU.md`
- `SKADI_DOCS_INTERNAL_RU.md`

Третий слой:

- внутренние contracts и drafts

## 9. Что можно сделать следующим шагом

- добавить `docs-en/` и начать реальные EN-версии ключевых страниц;
- завести GitHub Action на сборку сайта;
- подключить публикацию на GitHub Pages;
- затем интегрировать переводческий сервис.
